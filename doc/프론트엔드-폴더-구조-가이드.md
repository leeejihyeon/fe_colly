# 프론트엔드 폴더 구조 가이드

## 1. 목표
- 백엔드와 동일하게 기능 확장 시에도 유지보수가 쉬운 구조를 유지한다.
- 도메인(기능) 단위로 독립 개발이 가능하도록 구성한다.
- UX 상호작용(바텀시트, 터치 피드백)도 공통 규칙으로 관리한다.

## 2. 권장 구조
```text
src
├─ app
│  ├─ AppRoot.tsx
│  ├─ providers
│  └─ navigation
├─ features
│  ├─ community
│  │  ├─ screens
│  │  ├─ components
│  │  ├─ api
│  │  └─ types
│  ├─ experience
│  └─ mypage
└─ shared
   ├─ config
   ├─ ui
   ├─ theme
   ├─ lib
   └─ types
```

## 3. 폴더 역할
- `app`
  - 앱 부트스트랩, 전역 Provider, 루트 네비게이션을 둔다.
- `features`
  - 도메인(기능)별 화면/컴포넌트/API 타입을 둔다.
  - 다른 feature 내부 구현을 직접 참조하지 않는다.
- `shared`
  - 2개 이상 feature에서 공통으로 쓰는 UI/유틸/타입만 둔다.
- `shared/config`
  - 실행 환경(`dev/stage/prod`)과 API 베이스 URL 규칙을 관리한다.

## 4. 의존 방향 규칙
- `app -> features + shared`
- `features -> shared`
- `shared`는 `app`/`features`에 의존하지 않는다.

## 5. 네이밍 규칙
- 화면: `SomethingScreen.tsx`
- 공용 레이아웃: `SomethingContainer.tsx`
- API 래퍼: `somethingApi.ts`
- 타입: `something.ts`, `api.ts`

## 6. 현재 적용 사항
- React Navigation Bottom Tabs 도입
  - 파일: `src/app/navigation/RootNavigator.tsx`
- 인증/진입 UX 정책
  - 앱 첫 진입은 `Community` 탭을 기본으로 한다.
  - 로그인은 비로그인 사용자가 글쓰기/참여 신청/상세 상호작용 등 보호 액션을 시도할 때 노출한다.
  - 즉, `선탐색 후 로그인` 흐름을 기본 정책으로 유지한다.
- 환경값 분리
  - 파일: `src/shared/config/env.ts`
- Community API 실제 호출
  - 파일: `src/features/community/api/communityApi.ts`
  - 화면: `src/features/community/screens/CommunityScreen.tsx`
- 커뮤니티 위치 선택 UX
  - `Location` 탭 시 국가/도시 선택 바텀시트 표시
  - 백드롭은 즉시 표시, 바텀시트만 아래→위 슬라이드
- 터치 피드백(햅틱)
  - 주요 버튼/필터/선택 칩에 `onPressIn` 진동 피드백 적용

## 7. 실행 규칙
- Metro는 항상 먼저 실행한다.
  - `npm start`
- iOS 실행
  - `npx react-native run-ios`
- Android 실행
  - `npx react-native run-android`

## 8. 확장 계획
- 커뮤니티 상세/작성/참여 플로우를 `features/community` 내부로 확장한다.
- 체험/마이페이지도 동일 패턴(`screens/components/api/types`)으로 확장한다.
- 햅틱 유틸을 `shared/lib`로 분리해 전 화면에 일관 적용한다.
